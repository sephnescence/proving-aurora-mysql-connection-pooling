# Design Document

Consolidated reference for all architectural and implementation decisions made across the planning Q&A.

---

## Project Goal

Prove connection pooling in Amazon RDS Aurora for MySQL using RDS Proxy (server-side pooling) versus a direct Aurora connection (no pooling), observed via `information_schema.processlist` under sequential and concurrent load.

## Infrastructure

### AWS Region

`ap-southeast-2`

### CloudFormation Stacks

| Stack name                     | Deploy method               | Contents                                                        |
| ------------------------------ | --------------------------- | --------------------------------------------------------------- |
| `aurora-pooling-poc-bootstrap` | Manual (AWS CLI once)       | Deploy IAM role, ECR repos (OIDC provider is pre-existing account infrastructure, not managed here) |
| `aurora-pooling-poc`           | `deploy.yml` GitHub Actions | Everything else (VPCs, EKS, Aurora, RDS Proxy, IAM, CloudWatch) |

### VPC Topology

**VPC A — `10.0.0.0/16`**

- Internet Gateway + NAT Gateway (public subnets)
- EKS worker nodes (private subnets)
- Pooled Aurora Serverless v2 cluster (private subnets)
- RDS Proxy (private subnets)

**VPC B — `10.1.0.0/16`**

- Unpooled Aurora Serverless v2 cluster (private subnets)
- No internet gateway or NAT gateway required

**VPC Peering** between A and B, with route tables and security groups allowing EKS nodes (VPC A) to reach the unpooled Aurora cluster (VPC B) on port 3306.

### EKS Cluster

| Setting            | Value                    |
| ------------------ | ------------------------ |
| Name               | `aurora-pooling-poc-eks` |
| Kubernetes version | 1.32                     |
| Node type          | t3.small (1 node)        |
| Node location      | VPC A private subnets    |
| Namespace          | `pooling-poc`            |

### Aurora Clusters (both)

| Setting         | Value                                                       |
| --------------- | ----------------------------------------------------------- |
| Engine          | Aurora MySQL 3.x (MySQL 8.0 compatible)                     |
| Type            | Serverless v2 (0.5–1 ACU)                                   |
| Master username | `admin`                                                     |
| Master password | Auto-generated by CloudFormation, stored in Secrets Manager |
| Database name   | `pooling_poc`                                               |

### RDS Proxy (pooled stack only)

Sits in VPC A private subnets. References the pooled Aurora cluster's Secrets Manager secret for Aurora authentication. Default pool settings: `MaxConnectionsPercent=100`, `MaxIdleConnectionsPercent=50`.

Pooled NestJS app connects to the RDS Proxy endpoint. Unpooled NestJS app connects directly to the Aurora cluster endpoint via VPC peering.

### Application Load Balancer

Created by AWS Load Balancer Controller from Kubernetes Ingress. Internet-facing (public, for Artillery access from GitHub Actions). Path-based routing:

- `/pooled/*` → pooled-app service; ALB **strips the prefix** before forwarding (`/processlist`)
- `/unpooled/*` → unpooled-app service; ALB strips the prefix before forwarding

### IAM Roles

| Role                     | Location        | Trusts      | Purpose                                                          |
| ------------------------ | --------------- | ----------- | ---------------------------------------------------------------- |
| Deploy role              | Bootstrap stack | GitHub OIDC | `deploy.yml`, `teardown.yml`, `teardown-bootstrap.yml` workflows |
| Artillery role           | Main stack      | GitHub OIDC | `artillery-*.yml` workflows; read CF outputs + CloudWatch Logs   |
| ESO IRSA role            | Main stack      | EKS OIDC    | External Secrets Operator reads Secrets Manager                  |
| ALB Controller IRSA role | Main stack      | EKS OIDC    | AWS Load Balancer Controller creates/manages ALBs                |
| Fluent Bit IRSA role     | Main stack      | EKS OIDC    | Fluent Bit writes logs to CloudWatch                             |

All roles follow principle of least privilege. Artillery role has narrower permissions than deploy role (CloudFormation:DescribeStacks, CloudWatch Logs read only).

### Helm Installs (in deploy workflow, after CloudFormation)

1. **External Secrets Operator** — namespace `external-secrets-system`
2. **AWS Load Balancer Controller** — namespace `kube-system`
3. **Fluent Bit** (`aws-for-fluent-bit`) — namespace `amazon-cloudwatch`; routes `pooling-poc` pod logs to per-app CloudWatch log groups based on pod labels

### GitHub Secrets Required

Only one secret needs to be created in GitHub Repository Secrets:

- `AWS_ACCOUNT_ID` — 12-digit AWS account number

OIDC handles authentication; no access keys needed.

### `@pooling-poc/db-query` (shared NestJS module)

**Exports:** `DbQueryModule`, `DbQueryService`

**Service interface:**

```typescript
processlist(testType: string): Promise<ProcesslistRow[]>
ping(): Promise<void>  // used by health check
```

**ProcesslistRow type:**

```typescript
{
  ID: number
  USER: string
  HOST: string
  DB: string
  COMMAND: string
  TIME: number
  STATE: string
  INFO: string | null
}
```

**Query:**

```sql
SELECT ID, USER, HOST, DB, COMMAND, TIME, STATE, INFO
FROM information_schema.processlist
```

**Logging:** `nestjs-pino` (JSON output). Each `processlist()` call logs:

```json
{ "message": "processlist_result", "testType": "...", "rowCount": N, "rows": [...] }
```

**Connection strategies (controlled by `DB_CONNECTION_MODE`):**

- `persistent` — single long-lived connection, created on module init, destroyed on module destroy. Used by pooled-app (connecting to RDS Proxy).
- `per-request` — new mysql2 connection created and closed for every `processlist()` call. Used by unpooled-app (connecting directly to Aurora). This causes each request to open a visible connection in Aurora's PROCESSLIST, which is what we want to prove.

`ping()` always creates a temporary connection regardless of `DB_CONNECTION_MODE`.

**SSL:** When `DB_SSL=true`, mysql2 is configured with `ssl: { rejectUnauthorized: false }`. Suitable for private VPC use.

**Unit tests (Vitest):**

1. Happy path — mysql2 returns rows; service returns them and logger is called with correct fields
2. Empty result — mysql2 returns `[]`; service returns `[]` and logger is called
3. Error — mysql2 throws; service propagates the error

### `@pooling-poc/pooled-app` and `@pooling-poc/unpooled-app`

Both apps are structurally identical. Differences are in environment variables only.

**Endpoints:**

- `GET /processlist` — reads `X-Test-Type` header (returns **400** if missing), calls `dbQueryService.processlist(testType)`, returns `ProcesslistRow[]` as JSON
- `GET /health` — `@nestjs/terminus`, checks HTTP liveness **and** DB connectivity (calls `dbQueryService.ping()`)

**Kubernetes readiness probe:** `GET /health`, initial delay 30s, interval **10 minutes**. Long interval prevents probe-driven connections from appearing as noise in PROCESSLIST during Artillery runs.

### `@pooling-poc/integration-tests`

Vitest tests using native `fetch` (Node 22 built-in). Runs against Docker Compose.

Calls `GET /processlist` on both apps with `X-Test-Type: integration` header.

**Assertions per app:**

1. HTTP status 200
2. Response body is a JSON array
3. Array has at least one row
4. Each row contains fields: `ID`, `USER`, `HOST`, `DB`, `COMMAND`, `TIME`, `STATE`, `INFO`

### `@pooling-poc/load-tests`

**Artillery configs:**

- `sequential.yml` — 3 virtual users, each making 5 sequential requests. Sends `X-Test-Type: sequential`. Target URL from `{{ $env.TARGET_URL }}`.
- `concurrent.yml` — 15 virtual users each making 1 simultaneous request. Sends `X-Test-Type: concurrent`.

**CloudWatch polling script** (`poll.ts`, run via `tsx`):

- Accepts test start timestamp, log group names, and `testType` as inputs
- Polls CloudWatch Logs Insights every **30 seconds** for **5 minutes**
- Query:
  ```
  fields @timestamp, testType, rowCount, rows
  | filter testType = "sequential"   (or "concurrent")
  | sort @timestamp desc
  | limit 50
  ```
- Console output per iteration: formatted raw log entries table + average `rowCount` across all results (connection count summary)
- AWS SDK: `@aws-sdk/client-cloudwatch-logs`

## Observability

- One for pooled-app, one for unpooled-app
- Fluent Bit routes logs based on pod labels in the `pooling-poc` namespace
- All log entries are JSON (nestjs-pino); filterable by `testType` field

### CloudWatch Dashboard

One dashboard with four Logs Insights widgets (time range: last 1 hour):

| Widget                | Log group              | Filter                    |
| --------------------- | ---------------------- | ------------------------- |

| Unpooled — sequential | unpooled-app log group | `testType = "sequential"` |

## GitHub Actions Workflows

All `workflow_dispatch` workflows are restricted to the `main` branch.

### `ci.yml`

**Triggers:** push (all branches), `workflow_dispatch`, schedule `0 2 * * *` (noon Brisbane, UTC+10)

**Steps:**

1. `pnpm install`
2. Run unit tests: `pnpm --filter @pooling-poc/db-query test`
3. Build Docker images for pooled-app and unpooled-app
4. `docker compose up -d`
5. Poll `GET /health` on both apps until ready (or timeout)
6. Run integration tests: `pnpm --filter @pooling-poc/integration-tests test`
7. `docker compose down` (always runs)
8. Fail build if step 2 or 6 failed

### `deploy.yml`

**Triggers:** `workflow_dispatch` (main only)

**Steps:**

1. Configure AWS credentials via OIDC (deploy role)
2. ECR login
3. Build and push pooled-app image (`{ecr_uri}:{git_sha}`)
4. Build and push unpooled-app image
5. Deploy CloudFormation stack `aurora-pooling-poc` (create or update)
6. `aws eks update-kubeconfig --name aurora-pooling-poc-eks --region ap-southeast-2`
7. Helm install: ESO
8. Helm install: AWS Load Balancer Controller
9. Helm install: Fluent Bit (with log group routing config)
10. Query stack outputs → set env vars for envsubst
11. `envsubst < k8s/<manifest> | kubectl apply -f -` for all manifests
12. Wait for deployments to be ready

### `teardown.yml`

**Triggers:** `workflow_dispatch` (main only)

**Steps:**

1. Configure AWS credentials via OIDC (deploy role)
2. `aws eks update-kubeconfig`
3. `kubectl delete ingress,deployment,service --all -n pooling-poc`
4. Poll until ALB ENIs are released (VPC subnets clear)
5. `aws cloudformation delete-stack --stack-name aurora-pooling-poc`
6. Wait for stack deletion to complete

### `teardown-bootstrap.yml`

**Triggers:** `workflow_dispatch` (main only)

**Steps:**

1. Configure AWS credentials via OIDC (deploy role)
2. Guard: check `aurora-pooling-poc` stack does **not** exist — fail with error if it does
3. Query bootstrap stack outputs for ECR repo names
4. Empty both ECR repositories (delete all images)
5. `aws cloudformation delete-stack --stack-name aurora-pooling-poc-bootstrap`

### `artillery-sequential.yml`

**Triggers:** `workflow_dispatch` (main only)

**Steps:**

1. Configure AWS credentials via OIDC (Artillery role)
2. Query `aurora-pooling-poc` stack outputs: ALB URL, pooled log group, unpooled log group
3. `pnpm install --filter @pooling-poc/load-tests`
4. Record test start timestamp
5. Run Artillery: `artillery run sequential.yml` with `TARGET_URL` set
6. Run CloudWatch polling script: `tsx poll.ts --testType sequential --startTime {timestamp} --pooledLogGroup {name} --unpooledLogGroup {name}`

### `artillery-concurrent.yml`

**Triggers:** `workflow_dispatch` (main only)

Identical to `artillery-sequential.yml` with `concurrent.yml` config and `--testType concurrent`.

> **Note:** Artillery workflows require the `aurora-pooling-poc` stack to be deployed first (the Artillery IAM role is created by that stack).

## Bootstrap Deployment (Manual, Once)

Deploy `infra/bootstrap.yml` manually before any GitHub Actions workflows can run:

```bash
aws cloudformation deploy \
  --template-file infra/bootstrap.yml \
  --stack-name aurora-pooling-poc-bootstrap \
  --capabilities CAPABILITY_NAMED_IAM \
  --region ap-southeast-2
```

After deployment, create one GitHub Repository Secret:

- `AWS_ACCOUNT_ID`: your 12-digit AWS account number
