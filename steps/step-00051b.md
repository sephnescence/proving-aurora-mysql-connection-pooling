# Step 00051b — Bootstrap and deploy infrastructure manually

## Context

Before the deploy workflow (step 51) can run via GitHub Actions, one one-time AWS account prerequisite must be in place. Once that's done, you can either trigger the deploy workflow from GitHub or run the equivalent AWS CLI commands directly from your terminal to get the infrastructure up so you can continue with step 51c.

Note: the ALB Controller IAM policy is now created automatically by the deploy workflow if it doesn't exist — no manual step required.

## Action

### 1. Deploy the bootstrap stack (one-time)

```bash
aws cloudformation deploy \
  --template-file infra/bootstrap.yml \
  --stack-name aurora-pooling-poc-bootstrap \
  --parameter-overrides \
    GitHubOrg=sephnescence \
    GitHubRepo=proving-aurora-mysql-connection-pooling \
  --capabilities CAPABILITY_NAMED_IAM \
  --region ap-southeast-2
```

### 2. Configure the GitHub repository secret

In your GitHub repository go to **Settings → Secrets and variables → Actions** and add:

| Secret | Value |
|--------|-------|
| `AWS_ACCOUNT_ID` | Your 12-digit AWS account number |

### 3. Deploy the main stack and apply Kubernetes manifests

Either trigger `.github/workflows/deploy.yml` manually from the GitHub Actions tab, or run the local deploy script:

```bash
chmod +x scripts/deploy-local.sh
./scripts/deploy-local.sh
```

See `scripts/deploy-local.sh` for the full sequence — it handles ECR login, image builds, CloudFormation deploy, kubeconfig update, Helm chart installs, and manifest application.

## Expected result

EKS is running, both deployments are rolled out, and `kubectl get pods -n pooling-poc` shows both pods as `Running`.
