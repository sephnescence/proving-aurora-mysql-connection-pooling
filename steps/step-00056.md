## Project purpose

Prove connection pooling in Amazon RDS Aurora for MySQL using RDS Proxy (server-side pooling) versus a direct Aurora connection (no pooling), observed via `information_schema.processlist` under sequential and concurrent load.

## Key architectural decisions

- `DB_CONNECTION_MODE=persistent` on pooled-app (single long-lived mysql2 connection → RDS Proxy)
- `DB_CONNECTION_MODE=per-request` on unpooled-app (new connection per request → Aurora directly)
- Both apps are structurally identical; differences are environment variables only
- nestjs-pino for JSON structured logging; every processlist call logs `{ message, testType, rowCount, rows }`
- `X-Test-Type` header is required on `GET /processlist`; returns 400 if missing
- Readiness probe interval is 10 minutes to avoid probe connections appearing in PROCESSLIST during Artillery runs
- VPC A (10.0.0.0/16): EKS + pooled Aurora + RDS Proxy
- VPC B (10.1.0.0/16): unpooled Aurora only; reached from EKS via VPC peering
- All K8s manifests use envsubst placeholders substituted by deploy.yml from CF stack outputs
- Bootstrap CF stack must be deployed manually once before any GitHub Actions workflow can run
- Only one GitHub secret needed: `AWS_ACCOUNT_ID`

## TypeScript
