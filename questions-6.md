# Round 6 Clarifications & Challenged Assumptions

## Critical: Teardown Will Fail Without K8s Cleanup First

The ALB is **not created by CloudFormation** — it's created by the AWS Load Balancer Controller when the Kubernetes Ingress resource is applied. CloudFormation has no knowledge of it.

When `teardown.yml` runs `aws cloudformation delete-stack`, CloudFormation attempts to delete the VPC and its subnets. It will fail because the ALB has network interfaces (ENIs) still attached to those subnets. The stack deletion will hang or error.

The teardown workflow must first:
1. Run `kubectl delete ingress,deployment,service --all -n <namespace>` — this triggers the ALB Controller to delete the ALB
2. Wait for the ALB to be fully deleted (polling for ENI cleanup)
3. Then run `aws cloudformation delete-stack`

This means the teardown workflow also needs EKS cluster access (kubeconfig). The same deploy IAM role likely has sufficient permissions for this. Is this approach acceptable?

A: Yes

## Critical: EKS Log Forwarding to CloudWatch is Not Automatic

NestJS pods write to stdout/stderr. These logs do **not** automatically appear in CloudWatch — a log forwarder must be installed in the cluster.

The standard solution is **Fluent Bit** via the `aws-for-fluent-bit` Helm chart. This is another Helm install step in the deploy workflow (similar to ESO and the ALB Controller). It routes pod stdout to the CloudWatch log groups defined in the CloudFormation stack.

- Is adding Fluent Bit as a Helm install step in the deploy workflow acceptable?
- Note: without this, **no logs reach CloudWatch at all** — the dashboard widgets and Artillery polling script would have nothing to query.

A: Yes, please add Fluent Bit

## Critical: NestJS Needs a JSON Logger for Logs Insights

CloudWatch Logs Insights filters using field queries like `filter testType = "sequential"`. This only works if log entries are **structured JSON**. NestJS's default logger writes plain text (`[Nest] LOG [AppModule] ...`) which Logs Insights can't parse into fields.

A JSON logging library needs to be added. Two common options:

- **nestjs-pino** — wraps Pino (very fast, minimal config, widely used with NestJS, outputs JSON by default)
- **winston** with `nest-winston` — more configurable, larger footprint

Which do you prefer? (nestjs-pino is the simpler choice for this use case.)

A: NestJS-pino is fine

The MySQL container in Docker Compose will have very few active connections (just the NestJS connection itself), so the processlist may return only 1–2 rows. Proposed assertions:

1. HTTP response status is `200`
2. Response body is a JSON array
3. Array contains at least one row
4. Each row has the expected fields: `ID`, `USER`, `HOST`, `DB`, `COMMAND`, `TIME`, `STATE`, `INFO`

Is this acceptable, or should the bar be higher or lower?
